# 결과 파일 개선 계획

## 1. 문제 원인 심층 분석

### 1.1 실제 출력 내용

```
1. nn nn
2. nn nn
[AI, nnnnnnn nn nn nn nn]
```

- **"nn nn"**: 한글 문자가 깨져 표시됨
- **"nnnnnnn nn nn nn nn"**: 한글 본문이 동일하게 깨짐
- **"[AI, ...]"**: 키워드 기반 fallback 형식으로 추정 (예: `[AI, 스마트헬스케어 기반 내용 작성 예정]`)

---

### 1.2 근본 원인 (Root Cause)

#### A. 한글 폰트 미지원 (핵심 원인)

| 항목 | 현재 상태 | 문제 |
|------|----------|------|
| 폰트 | `Helvetica` (ReportLab 기본) | 한글(CJK) 글리프 없음 |
| 결과 | 한글 → `nn`, `□`, 빈 박스로 대체 | 문서 가독성 완전 상실 |

**추론**: Helvetica는 라틴 문자만 지원. 한글 유니코드(U+AC00~U+D7A3)에 해당 글리프가 없어 PDF 뷰어가 대체 문자(`n` 또는 `□`)로 표시.

#### B. PDF 오버레이 방식의 한계

```python
# document_builder.py 49행
c.setFont("Helvetica", 10)  # ← 한글 미지원
c.drawString(50, y, line[:100])  # ← 고정 좌표, 줄바꿈 없음
```

- **고정 좌표(50, y)**: 양식 레이아웃 무시, 모든 PDF에 동일 위치
- **줄바꿈 미처리**: `drawString`은 한 줄만 그리며, 100자 초과 시 잘림
- **원본 양식 덮어쓰기**: 템플릿의 기존 구조(제목, 구분선 등)를 덮어씀

#### C. 콘텐츠 품질 가능성

- Gemini JSON 파싱 실패 시 fallback `"[{keywords} 기반 내용 작성 예정]"` 사용
- 피드백 라운드 3, 5에서 `sections`를 반환해야 개선된 본문 적용
- 분석/검증 라운드(1,2,4)는 `sections` 미반환 → 기존 content 유지

#### D. 출력 경로 불일치

- 설정: `OUTPUT_DIR=./outputs` (기본)
- 사용자 경로: `D:\AI\doligo\result\`
- `.env`에서 `OUTPUT_DIR=./result`로 변경했을 가능성

| 순위 | 원인 | 영향도 | 수정 난이도 |
|------|------|--------|-------------|

## 2. 단계별 개선 계획

### Phase 1: 한글 출력 수정 (최우선)

**목표**: 한글이 PDF에 정상 표시되도록 함

| 단계 | 작업 | 상세 |
|------|------|------|
| 1.1 | 한글 폰트 등록 | ReportLab에 `registerFont`로 NanumGothic, Malgun, NotoSansKR 등 TTF 등록 |
| 1.2 | 폰트 fallback | 시스템에 한글 폰트 없을 경우 `reportlab.pdfbase.ttfonts` 또는 `fpdf2` 등 대안 검토 |
| 1.3 | Paragraph 사용 | `drawString` 대신 `Paragraph` + `<font>` 태그로 자동 줄바꿈 및 한글 렌더링 |

**의존성**: `reportlab` (기본), 한글 TTF 파일 (NanumGothic 등, 무료 배포 가능)

---

### Phase 2: 레이아웃 및 가독성

**목표**: 긴 텍스트 줄바꿈, 여백, 가독성 개선

| 단계 | 작업 | 상세 |
|------|------|------|
| 2.1 | Paragraph + wrap | `reportlab.platypus.Paragraph`로 `width` 지정, 자동 줄바꿈 |
| 2.2 | 여백/폰트 크기 | 섹션별 제목(14pt), 본문(10~11pt), 적절한 margin |
| 2.3 | 페이지 넘김 | 내용 초과 시 `PageBreak`로 새 페이지 생성 |

---

### Phase 3: 양식 존중 (선택)

**목표**: 원본 PDF 양식의 영역을 최대한 유지

| 단계 | 작업 | 상세 |
|------|------|------|
| 3.1 | AcroForm 필드 활용 | PDF에 폼 필드가 있으면 `update_page_form_field_values` 등으로 채우기 |
| 3.2 | 좌표 매핑 | 파싱 단계에서 섹션별 (x,y) 추출, 빌더에서 해당 위치에만 출력 |
| 3.3 | 템플릿 미리보기 | `/api/parse` 응답에 좌표 정보 포함, 사용자 확인 가능하게 |

---

### Phase 4: 콘텐츠 품질 보강

**목표**: fallback 사용 최소화, 피드백 결과 반영 확실화

| 단계 | 작업 | 상세 |
|------|------|------|
| 4.1 | JSON 파싱 강화 | 정규식으로 `{...}` 블록 추출, 부분 파싱 시도 |
| 4.2 | 로깅 | 파싱 실패 시 로그에 raw 응답 저장, 디버깅 용이 |
| 4.3 | 피드백 적용 검증 | 라운드 3, 5 응답에 `sections` 없을 경우 재요청 또는 이전 content 유지 명시 |

---

### Phase 5: 출력 경로 및 UX

| 단계 | 작업 | 상세 |
|------|------|------|
| 5.1 | OUTPUT_DIR 명시 | `.env.example`에 `OUTPUT_DIR=./outputs` 명시, `result` 사용 시 문서화 |
| 5.2 | 다운로드 파일명 | `output_{job_id}.pdf` 대신 `지원과제_생성결과.pdf` 등 사용자 친화적 이름 |

---

## 3. 구현 우선순위 요약

```
Phase 1 (필수) → Phase 2 (권장) → Phase 4 (권장) → Phase 3 (선택) → Phase 5 (선택)
```

**즉시 적용 권장**: Phase 1 (한글 폰트) — 이 단계가 해결되면 "nn" 현상이 사라짐.
